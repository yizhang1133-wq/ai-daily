🔴 MiniMax M3 开源：428B MoE + 1M 上下文 + 智能体原生

MiniMax 发布开源权重模型 M3，约 428B 总参数、23B 激活参数，已上架 HuggingFace。该模型在编码与智能体基准上表现突出：59.0% SWE-Bench Pro、66.0% Terminal Bench 2.1、34.8% SWE-efficiency、28.8% KernelBench Hard、74.2% MCP Atlas。采用 MiniMax 稀疏注意力将上下文窗口扩展至 1M token，支持原生多模态。同步上线 MiniMax Code 工具及 API 平台，权重与技术报告预计约 10 天后发布。

AI PM 视角：M3 的发布标志着中国大模型厂商开始从"追赶者"向"差异化定义者"转变。428B/23B 的 MoE 架构选择意味着 MiniMax 在推理成本与性能之间做出了激进的权衡——用巨大的稀疏参数池换取推理时的低激活成本，这对企业级部署极具吸引力。1M 上下文窗口不是营销数字，而是直接对标 Claude 的核心差异化场景（长文档分析、代码库理解）。更值得注意的三个信号：一是 SWE-Bench Pro 59% 的成绩已经逼近顶级闭源模型，说明开源模型的编码能力正在跨越"可用"阈值；二是 MCP Atlas 74.2% 表明该模型原生为 Agent 交互设计，不是"事后适配"；三是同步发布 Code 工具和 API 平台，MiniMax 正在学习 OpenAI 的"模型+平台"组合拳。对于产品经理而言，这意味着在选型时"国产=性能差"的刻板印象正在瓦解，但需关注其实际长上下文稳定性（1M token 的 KV Cache 管理仍是工程难题）。

AI Builder 视角：M3 的架构选择具有强烈的工程导向。428B 总参数/23B 激活的 MoE 比率约为 18.6:1，这是一个相当激进的稀疏度——作为对比，Mixtral 8x7B 的比率约为 8:1，Qwen2-57B-A14B 约为 4:1。更高的稀疏度意味着更复杂的负载均衡和路由机制，MiniMax 声称的"稀疏注意力"可能是解决长上下文 KV Cache 爆炸的关键创新。Builder 应关注两点：一是 10 天后发布的技术报告，路由机制和学习策略将是复现关键；二是 1M 上下文的实际推理延迟和内存占用，这将决定其是否能在消费级硬件上运行。同步上线的 Code 工具如果支持本地部署，将是开发者首选的试验场。

研究型 Builder 视角：M3 的基准成绩需要审慎解读。SWE-Bench Pro 59% 是一个令人印象深刻的数字，但必须检查其训练数据污染情况——MiniMax 是否使用了 SWE-Bench 相关的 GitHub 仓库进行预训练或微调？MCP Atlas 74.2% 更值得研究者关注，因为它测试的是模型在标准化 Agent 协议下的交互能力，这比单一编码基准更能反映"模型作为智能体核心"的潜力。稀疏注意力机制的技术细节是最大看点：MiniMax 如何在保持 1M 上下文有效性的同时控制注意力计算的二次方复杂度？是线性注意力、滑动窗口、还是某种混合方案？这些问题将决定该架构的可扩展性上限。

可落地的方法论：- **模型选型新框架**：将"开源vs闭源"的二元选择升级为"场景-成本-可控性"三维矩阵，国产开源模型在长上下文和编码场景已具备替代可行性 - **Agent原生设计**：在产品设计阶段即要求模型支持 MCP/Function Calling 等 Agent 协议，而非后期通过提示工程 hack - **MoE部署预研**：关注稀疏模型的 Serving 基础设施（如 vLLM 对 MoE 的支持进展），提前评估路由负载均衡对延迟的影响 ---

02TH

🔴 Kimi K2.7-Code 开源：推理效率提升 30%，代码能力全面跃升

Kimi 发布并开源最新代码模型 Kimi-K2.7-Code。相比 K2.6，其在 Kimi Code Bench v2 上提升 +21.8%，Program Bench 提升 +11.0%，MLS Bench Lite 提升 +31.5%。推理效率改进，推理 token 使用量降低 30%，长时编码任务中指令遵循和端到端成功率均提升。6x 高速模式即将推出，即日起可通过 Kimi API 和 Kimi Code 使用。

AI PM 视角：K2.7-Code 的发布与 MiniMax M3 形成了有趣的"国产开源组合拳"——同一天，两家中国顶级实验室分别发布了通用 MoE 模型和专用代码模型。Kimi 的选择更聚焦：不做全场景覆盖，而是在代码这一垂直领域做到极致。+31.5% 的 MLS Bench Lite 提升和 30% 的推理 token 降低是一个"性能+效率"的双重胜利，意味着用户不仅得到更好的代码建议，还支付更少的费用（按 token 计费场景）。6x 高速模式的预告表明 Kimi 正在构建"质量-速度"双档位产品策略，这直接对标 Claude 的扩展思考模式。产品经理应关注 Kimi Code 产品的实际体验：如果其 IDE 集成达到 Cursor/Copilot 级别，配合开源模型的可定制性，将形成对西方编程 Agent 的实质性竞争。

AI Builder 视角：30% 的推理 token 降低是一个关键工程指标。在代码生成场景中，输出 token 数量通常远大于输入（一个函数实现可能消耗数百 token），30% 的降低意味着显著的成本节省和延迟改善。这通常通过更高效的推理策略（如投机解码、早期退出）或更紧凑的模型架构实现。Builder 应立即在自身代码库上测试 K2.7-Code 的实际表现，特别是长时编码任务中的指令遵循能力——这是从"代码补全"进化到"代码架构设计"的关键分水岭。Kimi API 的接入成本如果是开源模型水平，将是一个极具吸引力的替代方案。

研究型 Builder 视角：Kimi Code Bench v2 作为一个内部基准，其 +21.8% 的提升需要交叉验证。研究者应关注 Kimi 是否发布了该基准的详细构成和评测方法，以及模型在不同编程语言、不同复杂度任务上的分布表现。推理 token 降低 30% 的技术路径是核心研究问题：是通过蒸馏获得更紧凑的分布？通过改进的解码策略减少重复生成？还是通过架构优化（如更深的层但更窄的宽度）？Kimi 在此前的 K2.5 中展示了强大的长上下文能力，K2.7-Code 是否继承了这一优势、能否处理跨文件的代码库级任务，是判断其"研究价值"的关键。

可落地的方法论：- **垂直模型优先策略**：在代码、法律、医疗等特定领域，优先评估专用开源模型而非通用闭源模型，性能与成本比通常更优 - **效率基准纳入选型**：将"每美元生成的有效代码行数"作为核心指标，而非仅关注 top-line 准确率 - **双模式产品设计**：借鉴 Kimi 的"标准+高速"档位，为不同用户场景提供质量与速度的显式权衡 ---

🔴 字节豆包"任务模式"上线：全链路Agent执行与三级定价体系

6月12日，字节跳动旗下AI应用豆包大范围上线"任务模式"，支持定时执行、零代码网页生成、一键PPT生成、数据可视化分析等全链路Agent执行。原"思考模式"升级为"专家模式"，调用豆包大模型2.0 Pro版本，强化深度推理能力。App顶部模式切换改为"快速、专家、任务"。基础功能免费，高阶服务付费，专业版三档：标准版68元/月或688元/年，加强版200元/月或2048元/年，专业版500元/月或5088元/年。

AI PM 视角：豆包的"三模式"架构（快速、专家、任务）是一个经过深思熟虑的产品分层设计。快速模式覆盖日常闲聊和简单查询，专家模式对标深度推理和研究任务，任务模式则是向 Agent 执行的关键跃迁。这种分层的本质是"按认知负荷定价"——用户为不同复杂度的 AI 交互支付不同费用。68-500元/月的定价区间直接对标 ChatGPT Plus/Pro（约140-1400元/月），但豆包的基础功能免费策略意味着其用户漏斗顶部更宽。最值得关注的信号是"定时执行"——这是从"用户发起"到"AI主动"的范式转换，Agent 开始具备"预约"和"持续监控"能力。对于产品经理，这提出了一个新问题：当 AI 可以自主定时执行任务时，产品的"会话"概念是否需要被"工作流"取代？

AI Builder 视角：豆包的"任务模式"代表了中国 AI 应用层在 Agent 工程化上的领先探索。零代码网页生成和一键PPT生成不是新功能，但将其整合到统一的"任务"框架中意味着底层有一个统一的 Agent 编排层在协调多个工具（网页生成器、PPT引擎、数据可视化库）。Builder 应关注其"定时执行"的实现机制：是简单的 Cron 式调度，还是具备条件触发和错误恢复的智能调度？豆包2.0 Pro作为专家模式的底座，其推理能力是否足以支撑复杂的多步骤任务规划？这些技术细节将决定"任务模式"是营销概念还是真正的工程突破。

研究型 Builder 视角：豆包的产品演进提供了一个关于"AI应用如何从 Chatbot 进化为 Agent 平台"的鲜活案例。研究视角下最有趣的是"模式切换"设计——快速/专家/任务三种模式本质上是将同一个底层模型包装为不同的交互范式。这与 Anthropic 的"Artifacts"、OpenAI 的"Canvas"属于同一设计空间，但豆包更进一步加入了时间维度（定时执行）。研究者应关注用户在这三种模式间的切换频率和流失率：如果用户大量使用任务模式但很少使用专家模式，可能意味着"执行"需求正在超越"推理"需求，这将重塑模型能力优先级的研究方向。

可落地的方法论：- **模式分层设计**：将AI产品按"响应速度-推理深度-执行能力"三维分层，为不同用户场景提供显式模式切换 - **定时Agent预研**：评估产品中可被"定时化"的场景（报告生成、数据监控、内容发布），设计"人审后执行"和"完全自主"两级信任机制 - **定价锚定策略**：参考豆包的"免费+三档付费"结构，用基础功能建立用户习惯，用高阶功能（定时、多步骤、大文件）实现 monetization ---

🔴 Kimi K2.7-Code 开源：推理效率提升 30%，代码能力全面跃升

01

框架一：评估驱动开发（Evaluation-Driven Development, EDD）

今日事件：olmo-eval 工作台发布 + MiniMax M3 密集基准 + Kimi K2.7-Code 效率指标

在传统软件工程中，TDD（测试驱动开发）要求先写测试再写实现。在AI模型开发中，EDD 要求先定义评估协议再训练模型。olmo-eval 的"最小可检测效应"概念将这一理念推进到统计层面——不是"跑完基准看分数"，而是"设计实验验证假设"。MiniMax 在发布 M3 时同时公布 5 个编码基准和 1 个 Agent 基准，Kimi 公布效率提升 30% 的量化指标，都是 EDD 的实践案例。

落地方法：在AI产品团队中设立"评估工程师"角色，负责定义每个功能发布的可量化成功标准（准确率、延迟、成本、用户满意度），并在开发过程中持续监控是否达到"最小可检测效应"。

  • 追问 3：国产开源模型的"供应链安全"（训练数据合规性、模型权重完整性、长期维护承诺）应如何评估和审计？
