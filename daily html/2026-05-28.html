MCP 协议成为 AI 工具互操作的事实标准

Runway 正式推出 MCP 服务器，允许 Claude、ChatGPT、Cursor 等兼容 MCP 的 AI 智能体直接在对话中生成图像与视频，无需切换工作流；同一天 OpenAI 宣布产品支持私有 MCP 服务器的安全连接，允许企业在内部网络保留 MCP 服务器，通过仅出站 HTTPS 连接 ChatGPT、Codex 和 Responses API。

AI PM 视角：MCP（Model Context Protocol）正在从"一个不错的开源项目"变成**AI 基础设施的核心协议层**。Runway 和 OpenAI 同一天的动作形成了强烈的信号共振：前者证明 MCP 正在打通"创意生成"这一高价值场景，后者解决的是企业最敏感的"数据不出内网"顾虑。两者叠加意味着 MCP 已经跨越了"早期采用者"阶段，进入"主流落地"阶段。对产品经理而言，这预示着一个范式转移：未来的 AI 产品不再是独立的应用，而是**MCP 生态中的一个节点**——你的价值取决于你能被多少 Agent 调用，以及你在 Agent 工作流中的不可替代性。Claude Code 在同一天发布的 v2.1.152 中大幅增强技能系统（frontmatter 控制工具、/reload-skills、SessionStart 钩子等），进一步说明 Anthropic 也在将"可扩展的 Agent 能力"作为核心产品战略。

可落地的方法论：- **评估 MCP 接入优先级**：如果你的产品提供某种"能力"（生成、分析、查询、执行），问自己——用户是否希望在对话中直接调用这个能力？如果是，立即规划 MCP 服务器接入，窗口期约 3-6 个月，之后将成为标配而非差异化。 - **设计"Agent 原生"的产品架构**：不要只提供 API，要提供"可被 Agent 理解"的接口。这意味着清晰的工具描述、明确的输入输出 schema、可预期的错误处理，以及支持多轮上下文的状态管理。 ---

02TH

Cognition 成为全球最大独立智能体实验室（260 亿美元估值）

Cognition 宣布完成超 10 亿美元融资，估值达 260 亿美元，由 Lux Capital、General Catalyst 等领投。其企业使用量自年初增长超 10 倍，年化收入增至 4.92 亿美元。公司强调其拥有"首个编码智能体"等多项领先优势，并得到了 Peter Thiel 的重大投资。

AI PM 视角：这是 Agent 商业化进程中最具标志性的事件之一。260 亿美元估值和 4.92 亿美元 ARR 意味着什么？Devin 不是 demo，而是**已经产生近 5 亿美元年收入的商业产品**。年初至今使用量增长 10 倍，说明这不是早期尝鲜，而是企业客户的真实需求爆发。对 AI PM 而言，Cognition 的成功验证了两个关键判断：（1）**编程 Agent 是企业愿意为"自主交付"付费的第一个场景**——因为软件开发的ROI可量化、交付标准明确、迭代周期短；（2）"独立智能体实验室"这一定位说明资本市场正在将"Agent 原生公司"与传统 SaaS 区别对待，估值逻辑更接近平台型公司而非工具型公司。Peter Thiel 的投资也值得关注——他通常只投"从 0 到 1"的范式转移。

可落地的方法论：- **找到你的"Devin 场景"**：Cognition 的成功在于找到了"AI 独立交付完整工作产物"的第一个落地场景。你的产品领域是否存在"任务边界清晰、交付标准明确、人工复核成本低于完全人工执行"的场景？这是 Agent 商业化的最佳切入点。 - **设计"人机交接"的 SLA**：Agent 独立工作不意味着无人监督。Cognition 的 4.92 亿美元收入背后必然有一套成熟的"人类审查里程碑"机制。为你的 Agent 产品设计清晰的"交接点"——在哪里需要人类确认、在哪里可以自动继续。 ---

03TH

编程 Agent 生态全面战争：Claude Code v2.1.152 + Grok 登陆 Kilo IDE

Claude Code 发布 v2.1.152，核心改进包括 `/code-review --fix` 直接应用审查建议、技能系统增强（disallowed-tools、/reload-skills、SessionStart 钩子）、MessageDisplay 钩子事件等。同日，xAI 宣布 Grok 编程智能体登陆 Kilo IDE 平台，grok-build-0.1 可在 Kilo IDE 扩展或 CLI 中使用。

AI PM 视角：编程 Agent 赛道正在从"单一工具竞争"升级为"生态平台战争"。Claude Code 的更新透露了 Anthropic 的深层策略：通过**技能系统（Skills）**将 Claude Code 从一个"通用编程助手"变成"可扩展的编程平台"。frontmatter 控制工具权限、钩子系统定制会话行为、MessageDisplay 变换助手消息——这些功能都在为"第三方开发者基于 Claude Code 构建垂直编程 Agent"铺路。与此同时，Grok 选择 Kilo IDE 作为落地平台，走的是"模型 + IDE"的深度集成路线。两条路径各有优劣：Claude Code 追求"终端内的生态扩展"，Grok 追求"IDE 内的模型原生"。对 Builder 而言，这意味着**选择编程 Agent 工具链时，不仅要看模型能力，更要看其生态扩展性和工作流集成深度**。

可落地的方法论：- **采用"Plan-Phase-Review"方法论**：宝玉在同日分享的方法论极具实操价值——先用最强模型在多个平台的 Plan 模式下生成设计方案，选择最优方案拆分为 Phases 执行。对于复杂编程任务，这种"强规划 + 分阶段执行 + 人工审核"的模式比"一次性端到端生成"可靠性高一个数量级。 - **投资技能/插件系统**：如果你基于 Claude Code 或类似工具构建团队工作流，现在就开始将重复性的编程任务封装为 Skills。技能系统的 ROI 会随着团队规模线性增长。 ---

04TH

Qwen3.5 在 TokenSpeed 上达到 580 tps：开源模型推理性能新标杆

Qwen3.5 在 TokenSpeed 推理引擎上针对智能体工作负载达到创纪录的 580 tokens per second，由通义千问推理团队、LightSeek Foundation TokenSpeed 团队、NVIDIA 及 Mooncake 团队共同实现，采用 FlashAttention-4 优化。

AI PM 视角：580 tps 这个数字的意义需要放在"Agent 工作负载"的语境下理解。Agent 不是单次推理，而是**多轮、多工具调用、长上下文的连续推理**。在这种场景下，token 生成速度直接决定了 Agent 的"体感响应时间"——用户是否觉得 Agent 在"流畅地工作"还是"卡住思考"。Qwen3.5 的这一突破是多方协作的结果（通义千问 + TokenSpeed + NVIDIA + Mooncake），说明**开源推理栈正在形成完整的性能优化生态**。对产品经理而言，这意味着两件事：（1）开源模型的"可用性"正在快速逼近闭源模型，甚至在特定优化场景下超越；（2）推理性能的竞争正在从"模型层"扩展到"系统工程层"——FlashAttention、KV Cache 管理、调度策略等工程优化将成为新的竞争维度。

可落地的方法论：- **区分"模型能力"和"推理性能"**：在选择模型时，不要只看 benchmark 分数，要测试在你的实际工作负载（上下文长度、调用频率、并发量）下的端到端延迟。一个分数更高的模型，如果推理速度慢 3 倍，在 Agent 场景中的用户体验可能反而更差。 - **关注推理引擎选型**：TokenSpeed、vLLM、SGLang 等推理引擎的差异对你的产品延迟有实质性影响。如果你的产品已经规模化，投入 1-2 个工程师专门做推理优化，ROI 可能远高于模型切换。 ---

教皇 AGI 通谕 + Anthropic 零信任安全框架：AI 治理从边缘走向中心

教皇 Leo XIV 发布《Magnifica Humanitas》通谕，警告人工智能的使用绝非纯粹技术问题，当其进入影响人类生活的过程时，便触及权利、机会、地位与自由，Anthropic 联合创始人 Christopher Olah 出席发布；同日 Anthropic 发布针对企业部署自主 AI 智能体的安全框架，提出三层零信任架构及八阶段实施流程。

AI PM 视角：这两件事同时发生不是巧合，而是 AI 治理从"边缘议题"变为"核心议题"的标志。教皇通谕的特殊之处在于：（1）它将 AI 伦理提升到了**天主教社会教义**的高度，这意味着全球 13 亿天主教徒及其影响的政策制定者将以此为参照；（2）Christopher Olah 的出席说明顶尖 AI lab 正在主动参与宗教/哲学层面的对话，而非仅停留在技术讨论。与此同时，Anthropic 的零信任框架则提供了**可落地的企业安全蓝图**——从基础设施层到 Agent 行为层的三级防护，以及提示注入、工具投毒、记忆投毒等 Agent 特有威胁的应对。对产品经理而言，这意味着**安全不再是合规团队的"事后检查"，而是产品设计的"前置约束"**。

可落地的方法论：- **建立 Agent 安全的"威胁模型"**：参考 Anthropic 框架，为你的 Agent 产品设计最小可行的安全架构，至少覆盖：（1）提示注入过滤（输入侧）、（2）工具权限最小化（执行侧）、（3）记忆/上下文隔离（状态侧）。 - **将"可解释性"作为信任资产**：教皇通谕强调的权利、机会、自由等议题，本质上是"AI 决策是否可理解、可质疑、可申诉"。在产品的关键决策点提供"为什么这样决策"的解释，不仅是技术能力，更是合规能力和品牌信任资产。 ---

思考题四：当 AI 治理从"合规成本"变成"竞争优势"，你准备好了吗？

教皇通谕 + Anthropic 零信任框架 + 中国综合性立法，三重信号叠加说明 AI 治理正在从"可选项"变为"必选项"。请思考：

  • 你的产品中，哪些决策对用户有实质性影响（权利、机会、地位、自由）？这些决策点是否有可解释性设计？
  • 如果你的产品明天被要求提供"AI 决策审计日志"，你是否能在 24 小时内生成？
  • "透明"和"竞争力"之间是否存在张力？如何在保护商业机密的同时满足可解释性要求？
05

思考题五：开源模型的"可用性拐点"是否已经到来？

Qwen3.5 的 580 tps + 阿里云的全栈布局 + PyTorch 白金会员，说明开源生态正在形成"模型 + 推理引擎 + 云基础设施"的完整闭环。请思考：

  • 你的产品对闭源模型是否存在"过度依赖"？如果明天主力模型涨价 3 倍或停止服务，你的切换成本是多少？
  • 开源模型的推理性能优化（如 FlashAttention-4、TokenSpeed）是否足以覆盖你场景中的延迟要求？
  • 当开源和闭源模型的能力差距缩小到 10% 以内时，你的选型逻辑应该是什么？

01

路径 A：AI 产品经理进阶路径

如果你刚入门（0-6 个月）：

1. 理解 MCP 协议：阅读 MCP 官方文档（2 小时），尝试用一个现有工具（如 Claude Desktop）接入一个简单的 MCP 服务器（如文件系统或 SQLite），理解"工具描述 + schema + 调用"的基本模式。

2. 建立 Agent 产品思维：阅读 Anthropic 的《Building Effective Agents》和《Zero-Trust for AI Agents》，重点理解"Agent 不是更复杂的 prompt，而是有状态、有工具、有反馈循环的系统"。

3. 做一次端到端延迟拆解：选择你产品中的一个 AI 功能，用 Chrome DevTools 或后端日志拆解从用户点击到结果呈现的每一个环节，找出真正的瓶颈。

如果你已有经验（6-18 个月）：

1. 设计一个 MCP 服务器 PoC：选择你产品中最常被调用的一个能力，用 TypeScript 或 Python 实现一个最小可行的 MCP 服务器，让团队成员通过 Claude Code 调用它。记录"哪些设计决策是直观的、哪些是反直觉的"。

2. 深入研究编程 Agent 的 UX 模式：对比 Claude Code、Cursor、Codex、Grok 在以下维度的差异：任务启动方式（prompt vs 目标描述）、执行过程的可视化、人类介入点的设计、错误恢复机制。输出一份对比分析报告。

3. 建立 Agent 安全评估框架：参考 Anthropic 的零信任框架，为你的产品设计一个"最小可行安全评估表"，覆盖输入验证、权限控制、记忆隔离三个维度。

如果你已是资深 PM（18 个月以上）：

1. 研究 Agent 商业模式的演进：深入分析 Cognition、Sierra、Adept 等 Agent 原生公司的定价策略、收入结构和客户续约率，理解"按结果计费" vs "按使用量计费" vs "按席位计费"在不同场景的适用性。

2. 构建领域专属的 Agent 评估基准：为你的垂直场景设计 3-5 个"真实任务"（而非 benchmark 题目），建立内部评估体系。定期用最新模型测试，追踪"能力边界"的扩张速度。

3. 参与 AI 治理的对话：阅读教皇通谕《Magnifica Humanitas》的全文（或关键章节），结合你的产品思考"技术决策中的伦理维度"。这不是为了合规，而是为了建立产品的长期信任资产。

02

路径 B：AI Builder 技术进阶路径

如果你刚入门（0-6 个月）：

1. 掌握 MCP 服务器开发：用官方 SDK（TypeScript/Python）实现一个连接你常用 API 的 MCP 服务器。重点理解 tools/calls/resources 三种原语的语义差异。

2. 熟悉一种编程 Agent 工具链：选择 Claude Code 或 Cursor，用它完成一个真实的小项目（如一个内部工具或自动化脚本），记录"Agent 在哪些地方表现出色、哪些地方需要人工纠偏"。

3. 学习推理优化基础：了解 FlashAttention、KV Cache、Continuous Batching 的基本原理，知道它们如何影响你的应用延迟和成本。

如果你已有经验（6-18 个月）：

1. 优化你的推理栈：如果你正在使用 vLLM 或类似引擎，尝试升级到支持 FlashAttention-4 的版本，对比优化前后的延迟和吞吐量变化。记录不同上下文长度下的性能曲线。

2. 实现一个"自信度阈值"机制：在你的 Agent 中集成一个"当模型置信度低于 X 时暂停并请求人类确认"的机制。实验不同的阈值对准确率和用户体验的影响。

3. 构建多模型路由层：在你的系统中实现基于成本/质量/延迟的动态模型切换逻辑，至少接入两个模型供应商。模拟主模型故障时的降级体验。

如果你已是资深工程师（18 个月以上）：

1. 研究增量权重同步和分布式训练：深入阅读 Hugging Face TRL 的增量权重同步技术，理解万亿参数模型在分布式环境下的工程挑战。

2. 构建企业级 Agent 安全架构：参考 Anthropic 的零信任框架，在你的组织中实现一个三层防御原型（基础设施层 + 行为层 + 治理层），至少覆盖提示注入检测、工具权限最小化和记忆隔离。

3. 参与开源推理生态：向 TokenSpeed、vLLM、SGLang 等开源推理项目贡献代码或性能测试报告，深入理解开源推理栈的演进方向。
