OpenAI 内部报告：智能体 Codex 如何改变工作

OpenAI 在 2025 年 8 月至 2026 年 6 月间的内部观察显示，Codex 已取代 ChatGPT 成为主要工作工具，各部门输出 token 中 Codex 占比从不足 10% 升至 99.8%。80.6% 的个体用户曾发起预计等效人类工作时间超过 30 分钟的请求，70.2% 超过 1 小时，25.6% 超过 8 小时；99 百分位用户每日生成超过 60 小时的 agent turns。非开发者用户增长迅猛：个体用户增长 137 倍，组织用户增长 189 倍。Legal、Finance、Recruiting 等部门在 2026 年 4 月前后跨过 Codex 使用过半的拐点，平均每位律师或招聘人员超过 85% 的输出 token 来自 Codex。

AI PM 视角：这份报告是 Agentic AI 从"开发者玩具"跃迁为"企业核心生产力"的最硬证据。首先，它打破了两个刻板印象：一是 Agent 只适用于软件工程，二是非技术用户不会使用长任务 Agent。Legal、Finance、Recruiting 等部门在 2026 年 4 月前后同时跨过 50% 使用拐点，说明 Agent 正在覆盖所有以"结构化文档 + 规则推理 + 多步骤执行"为特征的知识工作。对于产品经理而言，这意味着产品设计的重心需要从"如何生成更好的一次性回答"转向"如何管理跨小时、跨系统、跨角色的持续任务"。 其次，报告中的"等效人类工作时间"指标值得关注。当 25.6% 的用户请求超过 8 小时人类等效工作时长时，产品形态已经发生质变：用户不再是在聊天框里提问，而是在委托一个虚拟同事。这要求 PM 重新设计权限模型、审计日志、中断恢复、人机交接和错误兜底机制。同时，组织层面的变更管理将成为产品成功的关键变量，因为 Agent 的采用速度可能超过员工的适应速度。

研究型 Builder 视角：这份报告的价值在于提供了真实世界、长周期、大样本的 Agent 使用数据，而不是实验室基准。它验证了长任务 Agent 在日常工作场景中的可行性和用户粘性，同时也提出了多个可研究的问题：是什么任务特性使得 Legal/Finance/Recruiting 同时跨过拐点？Agent 在哪些环节失败率最高？如何量化"agent turns"与实际业务产出之间的关系？对于研究型 Builder，这是一个绝佳的研究-to-product 案例：先有大模型能力，再有内部工具化实验，最后形成可产品化的 Agent 形态。

可落地的方法论：- 本周：在你的产品或组织内识别三类"重复性知识工作"场景（文档审查、数据整理、规则匹配），用现有 Agent 工具做一个 2 小时的端到端原型，记录任务成功率与人类复核比例。 - 本季度：为 Agent 产品设计一套"任务委托"用户体验框架，包含目标输入、进度可视化、关键决策点确认、异常回退和结果审计五个模块。 ---

02TH

IBM 首度推出亚纳米级芯片技术

IBM 于 2026 年 6 月 25 日发布全球首款亚纳米级芯片技术，采用 0.7 nm（7 埃米）节点与全新三维纳米堆叠（nanostack）架构。指甲盖大小的芯片集成近 1000 亿个晶体管，密度约为 IBM 2021 年 2 nm 芯片的两倍。相比 2 nm 芯片，性能最高提升 50%，能效最高提升 70%。纳米堆叠架构还实现 SRAM 面积缩减 40%，有助于支撑先进 AI 工作负载的高带宽需求。该技术已在 VLSI 2026 会议上验证，IBM 预计 5 年内量产。

AI PM 视角：对于 AI 产品经理，这则新闻的核心意义不在于芯片本身，而在于它预示的"算力成本曲线"和"模型能力上限"将同时被推高。0.7 nm 工艺配合三维堆叠，意味着同样功耗下可部署的模型规模可能翻倍，或者在端侧运行更大参数的模型。这会直接影响产品决策：未来 3-5 年内，本地推理、私有化部署、端侧 Agent 的可行性将显著提升。同时，SRAM 面积缩减 40% 对 Transformer 类模型尤为关键，因为注意力机制对片上缓存极其敏感。 从竞争格局看，IBM 此项技术若能如期量产，可能打破当前 AI 芯片由 NVIDIA 主导的单一格局，并为云服务厂商提供新的差异化基础。PM 在规划产品路线图时，应将"端侧大模型"和"混合云推理"作为高概率场景纳入考虑，而不是继续假设所有复杂推理都发生在云端。

研究型 Builder 视角：三维纳米堆叠架构是这项技术的真正亮点。传统的平面微缩已经接近物理极限，而垂直堆叠通过增加互连密度和缩短信号传输距离，在提升性能的同时降低功耗。研究型 Builder 应关注两个可验证的命题：一是这种架构对 Transformer 推理的实际加速比（不仅是峰值算力，还包括内存带宽和延迟）；二是 SRAM 缩减是否真的能在相同面积内支持更大的 KV Cache。VLSI 2026 会议论文是必读材料。

可落地的方法论：- 本周：梳理你产品当前依赖云端推理的功能模块，评估哪些可以在未来 2-3 年内迁移到端侧或混合部署。 - 本季度：建立"算力-成本-体验"三角评估框架，为每个 AI 功能预设云端、边缘、端侧三种部署方案的成本阈值。 ---

美国政府要求 OpenAI 暂缓 GPT-5.6 广泛发布

The Information 报道，美国政府因安全顾虑要求 OpenAI 暂缓 GPT-5.6 的广泛发布，改为推出受控预览版。OpenAI 计划先向小部分合作伙伴提供早期访问，并由政府逐客户审批准入。这一非常规做法主要担忧模型在自动化高技能网络工作上的能力：既能帮助防御者更快发现漏洞，也可能被攻击者用于加速测试漏洞利用。本周四，CEO Sam Altman 已向员工确认该审批流程。

AI PM 视角：这是 AI 产品发布逻辑的历史性转折点。过去，模型发布主要由公司内部的安全评估和市场节奏决定；现在，政府逐客户审批成为前置条件。这意味着"安全"不再是一个技术团队的 checklist，而是产品 Go-to-Market 的核心变量。对于产品经理，有三层启示：第一，产品路线图需要预留合规和审批缓冲期，不能再按纯技术迭代节奏承诺客户；第二，toB 产品的销售和客户成功团队需要具备解释安全审批流程的能力；第三，"受控预览版"可能成为高能力模型的标准发布形态，类似于新药临床试验的分阶段放行。 这一事件也与 OpenAI 自己的 Codex 报告形成张力：一方面 Agent 正在大幅提升生产力，另一方面最顶尖的模型能力因安全担忧被限速释放。PM 需要在"能力开放"与"风险可控"之间找到动态平衡，而这将成为下一代 AI 产品的核心设计约束。

研究型 Builder 视角：从研究角度，这则新闻提出了一个关于"能力-风险对应关系"的严肃问题：我们是否有足够的评估基准来量化模型在网络安全等领域的双重用途风险？目前的 SWE-Bench、网络安全 CTF 等基准主要衡量能力，但缺乏对"恶意使用潜力"的标准化度量。研究型 Builder 可以关注红队测试、对抗评估、能力阈值监测等方法论，这些都将成为高能力模型安全发布的必要基础设施。

可落地的方法论：- 本周：审查你产品中所有依赖前沿模型的功能，识别可能被归类为"双重用途"或"高技能自动化"的场景，并制定最小权限原则。 - 本季度：建立模型安全分级发布流程（内测 → 受控预览 → 定向开放 → 广泛发布），每级配套明确的能力评估、用例审查和回滚机制。 ---

Ornith-1.0 开源模型家族发布，专注 Agentic Coding 全参数规模

Ornith-1.0 开源模型家族发布，专注智能体编程（Agentic Coding），覆盖 9B Dense、31B Dense、35B MoE 及 397B MoE 全参数规模。在 Agent Coding 基准上达开源顶尖：SWE-Bench Verified 82.4、SWE-Bench Pro 62.2、Terminal-Bench 2.1 77.5、NL2Repo 48.2、SWE Atlas 41.2 QnA、ClawEval 77.1。基于 gemma4 和 qwen3.5 后训练，采用强化学习联合优化任务脚手架（scaffold）与最终解决方案，让模型自主改进执行框架。全系列 MIT 开源，提供 GGUF 版本。

AI PM 视角：Ornith-1.0 的发布标志着开源社区在 Agentic Coding 领域已经具备与顶级闭源模型正面对抗的能力。对于 AI PM，这是一个强烈的信号：如果你的产品依赖闭源模型提供编码 Agent 能力，未来 6-12 个月内需要重新评估供应商策略。MIT 开源 + GGUF 版本意味着企业可以在本地、私有云甚至端侧部署，这对金融、医疗、政务等敏感行业极具吸引力。 更值得关注的创新点是"强化学习联合优化任务脚手架与最终解决方案"。这不是简单的模型微调，而是让模型学会"如何组织自己的工作环境"。这意味着 Agent 的能力边界从"写代码"扩展到了"设计自己的工作流"。PM 在设计编码 Agent 产品时，应开始考虑如何让模型自主选择和组合工具，而不是由人类预先设定固定的工具链。

研究型 Builder 视角：Ornith 的技术 novelty 在于其后训练范式：同时优化 scaffold（任务执行框架）和 solution（最终方案）。这与传统的"先训练基础模型，再训练工具调用"的两阶段方法不同，它把"元认知"（如何选择和改进行动框架）也纳入了优化目标。研究型 Builder 可以复现其训练流程，验证在 SWE-Bench 等基准上的性能是否可稳定重现，以及 scaffold 优化在不同类型的编码任务中的迁移性如何。

可落地的方法论：- 本周：在你的编码 Agent 工作流中接入 Ornith-1.0（从 9B 或 31B 版本开始），对比当前闭源模型在 3-5 个真实任务上的成功率和成本。 - 本季度：建立内部编码 Agent 基准，包含单元测试生成、代码审查、Bug 修复、跨文件重构等任务，每季度用最新开源和闭源模型跑分。 ---

General Intuition 完成 3.2 亿美元融资，用游戏数据训练通用 AI 智能体

General Intuition 以 23 亿美元估值完成 3.2 亿美元融资，累计披露融资 4.54 亿美元。公司从旗下游戏剪辑平台 Medal 获取数亿小时含精确按键动作标签的游戏操作数据，训练单一模型同时驾驭 Fortnite 等虚拟环境和四足机器人。演示中，AI 智能体在游戏中连续运行 100 小时，机器人仅靠 8 分钟真实街道数据微调即可自主探索办公室。本轮由 Khosla Ventures 领投，General Catalyst、Jeff Bezos、Eric Schmidt 等参投。资金将用于通过 CoreWeave 扩大计算规模、预训练下一代模型，计划夏末前开放 API。

AI PM 视角：General Intuition 展示了一种全新的 AI 产品护城河：不是模型参数，而是"可交互环境 + 精确动作标签 + 跨域迁移能力"。游戏数据的价值在于它提供了物理世界难以获取的"大规模、低成本、可重复、带精确控制信号"的训练环境。对于产品经理，这开启了一个新的产品类别：从虚拟环境预训练、到真实世界微调的"通用智能体"。 这一模式对机器人、自动驾驶、工业自动化、物流等领域有深远影响。如果"8 分钟真实街道数据微调即可自主探索办公室"能够被稳定复现，那么机器人产品的部署成本将从"收集海量真实数据"转向"设计高效的仿真到现实迁移流程"。PM 需要关注这家公司的 API 开放进度，并思考自己的硬件产品是否可以接入这种预训练智能体能力。

研究型 Builder 视角：研究上的核心问题是：游戏环境中学到的行为表征，为什么能够迁移到真实世界的四足机器人？可能的机制包括：底层运动控制策略的泛化、视觉-动作关联的跨域不变性、以及任务目标的层次化表示。研究型 Builder 应关注其数据构成、模型架构和微调策略，尤其是"100 小时游戏连续运行"和"8 分钟真实数据微调"这两个数字背后的训练细节和失败案例。

可落地的方法论：- 本周：如果你从事机器人、自动驾驶或工业自动化领域，列出当前产品中数据采集成本最高的 3 个场景，评估是否可以用仿真预训练 + 真实数据微调的范式替代。 - 本季度：关注 General Intuition 的 API 发布，设计一个最小可行实验，测试其预训练模型在你目标场景中的零样本/少样本表现。 ---

3 CROSS-CUTTING MODELS
01

框架 1：Agentic 工作的"临界点"模型

今日的多条事件共同指向一个判断：Agent 正在跨越从"辅助工具"到"工作承担者"的临界点。

  • 触发条件：OpenAI Codex 内部报告显示，Legal、Finance、Recruiting 等部门同时跨过 50% 使用拐点，说明 Agent 的适用性不局限于技术岗位，而是覆盖所有"结构化输入 + 规则推理 + 多步骤执行"的知识工作。
  • 产品形态变化：Codex 移动端 GA、Runway Agent 2.0 和 Claude Code 的权限/可观测性增强，说明产品设计的重心正从"单次交互质量"转向"长任务委托、跨设备协同、审计与回退"。

产品启示：PM 在规划 Agent 产品时，应避免只关注"模型能做什么"，而应设计完整的"任务生命周期管理"：目标委托、进度可视化、关键决策确认、异常处理、结果审计。同时，需要为不同角色（开发者、营销人员、律师、招聘人员）设计差异化的 Agent 工作流，而不是一个通用聊天界面。

02

框架 2：开源-闭源-地缘的"三角张力"

今天的 AI 产业同时受到三条力量拉扯：开源模型能力逼近闭源、闭源厂商指控知识提取、以及政府安全审查干预发布节奏。

  • 开源逼近：Ornith-1.0 在 Agentic Coding 基准上达到开源顶尖，MIT 许可证降低企业采用门槛。
  • 闭源防御：Anthropic 指控阿里巴巴非法获取 Claude 能力，反映闭源厂商对模型蒸馏和逆向工程的担忧。
  • 政府介入：美国政府要求 OpenAI 对 GPT-5.6 实行受控预览和逐客户审批，说明最强模型能力的释放不再由市场单独决定。

产品启示：产品经理需要建立"模型供应链韧性"思维。核心 AI 能力不应单一依赖某个闭源模型，而应设计可切换的模型后端（通过 OpenRouter MCP 等工具实现），并评估开源替代方案、合规风险和地缘政治影响。同时，高能力模型的发布节奏将更多受监管影响，产品路线图需要预留审批缓冲。

03

框架 3：数据与硬件作为"底层基础设施"的护城河

General Intuition 的游戏数据、IBM 的 0.7 nm 芯片、以及 Google 的线性弹性缓存，看似属于不同领域，实则共同说明：AI 竞争的下一阶段正在向"数据获取能力"和"算力成本优化"下沉。

  • 数据基础设施：游戏数据带精确动作标签，成为训练通用智能体的稀缺资源。
  • 硬件基础设施：亚纳米芯片决定未来 5 年模型部署的成本曲线和形态。
  • 系统优化：线性弹性缓存等技术在软件层进一步榨取算力效率。

产品启示：PM 不应只关注模型层和应用层的创新，还应关注底层数据、硬件和系统优化对用户体验和成本的长期影响。例如，端侧大模型的可行性将直接影响隐私敏感场景的产品设计；仿真数据预训练 + 真实数据微调的模式可能降低机器人等硬件产品的数据成本。

5 DEEP QUESTIONS
01

思考题 1：当 Agent 能够承担数小时的人类等效工作，产品经理应如何重新定义"人机协作边界"？

Codex 报告显示 25.6% 的用户请求超过 8 小时人类等效工作时长，这意味着 Agent 正在从"回答问题"变为"承担责任"。这种转变对产品架构、用户体验和组织管理都提出了全新要求。

  • 产品策略层面：你的产品中哪些任务可以交给 Agent 独立完成，哪些必须保留人类在环？划分的标准是什么？
  • 团队执行层面：如何设计 Agent 的"交接协议"，确保在 Agent 无法继续时人类能够快速接管？
  • 风险机会层面：当 Agent 出错时，责任如何界定？产品应如何建立用户信任和可审计性？
02

思考题 2：开源 Agentic Coding 模型达到闭源水平后，DevTool 产品的护城河在哪里？

Ornith-1.0 等开源模型的崛起，加上 OpenRouter MCP 等模型路由基础设施，意味着模型能力本身正在商品化。

  • 产品策略层面：如果任何团队都能调用顶尖编码模型，你的 DevTool 产品的差异化是工作流、数据、集成深度还是用户体验？
  • 团队执行层面：你应投资自建模型，还是专注于在开源模型之上构建垂直工作流？
  • 风险机会层面：模型商品化会降低进入壁垒，但也可能引发价格战，你的商业模式是否足够 resilient？
03

思考题 3：政府安全审查成为模型发布前置条件，AI 产品如何在合规与创新之间取得平衡？

美国政府对 GPT-5.6 的受控预览要求，预示高能力模型的发布将不再完全由公司自主决定。

  • 产品策略层面：你的产品是否依赖最前沿的模型能力？如果是，应如何设计分阶段发布和用例审查机制？
  • 团队执行层面：安全、合规、产品、销售团队如何协同，才能既满足监管要求又不拖累创新节奏？
  • 风险机会层面：监管是否会反而成为头部公司的护城河？小公司如何应对合规成本上升？
04

思考题 4：游戏数据训练真实世界机器人，这种跨域迁移的可靠边界在哪里？

General Intuition 用游戏数据训练机器人并在少量真实数据微调后取得效果，但跨域迁移的可靠性仍是开放问题。

  • 产品策略层面：你的硬件/机器人产品是否可以利用仿真或游戏数据进行预训练？哪些场景适合，哪些不适合？
  • 团队执行层面：如何设计仿真到现实的迁移验证流程，确保产品在真实环境中的安全性和稳定性？
  • 风险机会层面：如果跨域迁移被证明在更广泛任务上可靠，哪些行业的数据成本结构将被颠覆？
05

思考题 5：AI Economy 报告显示收入增长迅速但多数公司无法量化影响，PM 应如何建立可衡量的 AI 价值指标？

报告指出 31% 的标普 500 公司在财报电话会提及 AI，但仅 20% 量化影响，说明 AI 价值证明仍是薄弱环节。

  • 产品策略层面：你的 AI 功能是否有一个清晰、可追踪的北极星指标？如果没有，应如何定义？
  • 团队执行层面：如何在产品上线前建立基线，并在上线后持续追踪 AI 带来的实际业务价值？
  • 风险机会层面：当 Token 价格下降刺激用量增长时，如何区分"真实价值提升"和"因便宜而产生的使用增加"？
