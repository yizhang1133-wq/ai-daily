OpenAI ChatGPT 语音最大规模升级：双向 AI 语音模型 Bidi 1 已上线测试

6 月 23 日起，部分 ChatGPT 网页端与 App 端用户在模型选择器中发现新增的「Bidi 1」双向 AI 语音模型。该模型支持边说边听，允许用户在模型回应过程中随时打断并发出新指令，例如让模型从 1 数到 10 时中途喊停并倒数。OpenAI 尚未正式发布，预计本周将扩大测试范围。

AI PM 视角：Bidi 1 的上线意味着 ChatGPT 正在从「 turn-based 语音转文字再转语音」的伪语音交互，跃迁到真正的实时双工语音对话。对 PM 而言，这不只是交互延迟的优化，而是产品形态的根本变化：语音交互从「搜索/问答」变成「陪伴、教练、实时协作」的入口。打断能力、边听边说能力、语音中的指令切换能力，直接决定了 AI 能否进入会议、客服、教育、心理咨询、游戏 NPC 等高频场景。 从竞争格局看，OpenAI 此举是对 Anthropic、Google Gemini Live、以及众多端侧语音 Agent 的防御性卡位。语音是移动端最自然的入口，谁掌握了低延迟、可打断、情感自然的双向语音，谁就掌握了下一代 AI 助手的默认调用方式。值得注意的是，Bidi 1 与标准语音、高级语音并列，说明 OpenAI 内部也在多路线并行，尚未确定最终的产品形态。PM 在做语音产品时应避免「一步到位」的幻想，优先验证「可被打断的实时对话」在自身场景中的用户价值。

AI Builder 视角：对 Builder 来说，Bidi 1 的核心工程信号是「端到端双向语音模型」正在成为头部平台的标配。传统的 ASR → LLM → TTS 三阶段流水线，在延迟、情感、打断响应上存在结构性瓶颈。Bidi 1 可能采用了原生音频 token 模型（类似 GPT-4o 的 audio-in/audio-out），将语音直接作为模型的输入输出模态。这意味着 Builder 需要重新评估自己的语音架构：是否还需要独立的 ASR/TTS 供应商？如何处理语音中的背景噪声、多说话人、情绪识别？更关键的是，双向语音对服务端流式工程、客户端音频 buffer 管理、VAD（语音活动检测）提出了极高要求，这些才是产品差异化的真正壁垒。

研究型 Builder 视角：Bidi 1 的技术 novelty 在于它可能将「打断」建模为模型本身的能力，而非通过外部 VAD + 重触发机制实现。如果用户可以在任意时刻插入新指令，模型必须同时具备：持续监听音频流、理解新指令的语义边界、快速切换任务状态、并生成连贯的新回应。这要求模型在音频 token 层面拥有类似文本模型的「上下文切换」能力。一个值得验证的假设是：Bidi 1 是否通过统一的音频-文本联合训练，让模型学会从音频流中识别「打断信号」并自动重置生成？如果是，这将揭示多模态模型在实时交互中的新机制。研究型 Builder 可关注 OpenAI 后续是否释放技术细节，尤其是音频 token 的表示方式、训练数据构造（真实对话 vs 合成打断数据）、以及评估指标（除了 WER，还包括打断响应延迟、任务切换准确率等）。

可落地的方法论：- **本周可做**: 如果你在做语音助手或客服产品，立即用 ChatGPT App 体验 Bidi 1（如已获得权限），记录 10 组真实对话中的打断场景，对比现有 ASR+LLM+TTS 方案在延迟、错误率、自然度上的差距，形成一份 1 页的语音体验差距分析。 - **本季度可做**: 在你的产品中引入「最小可用语音交互」试点，优先选择用户双手被占用、需要快速确认/打断的场景（如驾驶助手、运动陪练），验证双向语音是否显著提升任务完成率。 - **长期布局**: 将语音交互从「功能」升级为「产品入口」，重新设计信息架构，让语音成为跨场景调用的默认方式，而不是某个具体功能的补充。 ---

Qwen-AgentWorld 开源：让 Agent 学会「先预测，再行动」

通义千问发布首个原生语言世界模型 Qwen-AgentWorld，覆盖 MCP、Search、Terminal、SWE、Web、OS、Android 七大领域。模型基于超 1000 万条真实交互轨迹，经 CPT→SFT→RL 三阶段训练，在 AgentWorldBench 上超越 GPT-5.4 与 Claude Opus 4.8。该模型既可作为解耦环境模拟器用于 Sim RL，也可作为智能体基础模型迁移到七个基准任务。

AI PM 视角：Qwen-AgentWorld 的出现标志着 Agent 训练范式的重大转变：从「让模型直接试错」到「让模型先在虚拟世界中预测行动后果」。对 PM 而言，这意味着未来构建 Agent 的成本结构可能发生质变——真实环境的试错成本（调用 API、操作真实系统、用户投诉）可以通过世界模型部分替代。尤其是在客服、运维、金融交易、工业控制等高风险场景，Sim RL 让 Agent 在上线前就能完成大量安全验证。 这一发布也揭示了开源阵营在 Agent 基础设施上的新打法：不再单纯追赶闭源模型的聊天能力，而是在「Agent 专用模型 + 训练数据 + 评测基准」上建立完整生态。Qwen-AgentWorld 同时开源了模型、训练方法、评测基准，这种「全栈开源」策略对中小团队极具吸引力。产品层面，它降低了构建垂直领域 Agent 的门槛：企业可以用自己的业务数据训练一个私有世界模型，再让 Agent 在其中进行安全演练。 competitive dynamics 上，这可能是中国模型公司在 Agent 时代实现差异化的一次重要尝试。

AI Builder 视角：Builder 需要关注 Qwen-AgentWorld 的两个技术范式：作为环境模拟器和作为 Agent 基础模型。前者意味着你可以用它替代部分真实环境进行 RL 训练，显著降低训练成本；后者意味着 LWM（Language World Model）的预训练知识可以迁移到新的 Agent 任务。工程上，这要求团队重新审视 Agent 架构：是否还需要手动编写复杂的 state machine？是否可以将部分决策逻辑交给世界模型进行「想象-验证」？同时，1000 万条真实交互轨迹的训练数据规模提示我们：高质量的 Agent 训练数据可能比模型架构本身更重要。Builder 应开始积累自己业务领域的交互日志，并考虑如何将其转化为可训练的轨迹数据。

研究型 Builder 视角：Qwen-AgentWorld 的技术 novelty 在于「原生语言世界模型」的规模化训练。传统世界模型多集中于游戏或机器人领域（如 World Models、Dreamer），而 Qwen-AgentWorld 将世界模型扩展到通用数字 Agent 环境（MCP、Web、OS 等）。研究型 Builder 应关注：1）CPT→SFT→RL 三阶段训练中，每一阶段对世界模型能力的具体贡献是什么？2）在 WideSearch 上 Sim RL 超越真实环境 RL 的现象，是否说明世界模型在某些任务上已经比真实环境更适合训练？3）LWM 预热迁移到未见过基准的能力，是否暗示了一种新的 meta-learning 机制？这些问题的答案将决定世界模型能否成为下一代 Agent 的标配组件。

可落地的方法论：- **本周可做**: 阅读 Qwen-AgentWorld 技术报告/论文，梳理其支持的七大环境哪些与你的业务最接近，评估是否可以引入作为 Agent 训练或测试环境。 - **本季度可做**: 选择一个低风险、高频率的 Agent 任务（如数据查询、报表生成），尝试用 Qwen-AgentWorld 或类似世界模型进行离线模拟训练，对比真实环境 RL 的成本与效果。 - **长期布局**: 建立企业级的「Agent 数字孪生」能力，将核心业务流程的交互日志结构化，为未来训练私有世界模型积累数据资产。 ---

Gemini 3.5 Flash 中的计算机使用

Google 将 Computer Use 作为内置工具集成至 Gemini 3.5 Flash，使开发者能构建跨浏览器、移动端和桌面环境的智能体。此前该功能仅作为独立模型在 Gemini 2.5 中提供，现已原生整合至主 Flash 模型。开发者可通过 Gemini API 及 Gemini Enterprise Agent Platform 调用，并可选企业级安全保护。

AI PM 视角：Computer Use 从独立模型变成 Flash 系列的内置工具，意味着「让 AI 操作计算机」正从实验能力变成标准化 API。对 PM 而言，这是 Agent 产品化进程中的关键里程碑：开发者不再需要自建复杂的视觉-动作协调系统，只需调用 Gemini API 就能让 Agent 跨浏览器、移动端、桌面执行任务。这将显著降低企业自动化、RPA 替代、跨应用知识工作的落地门槛。 更值得关注的是安全设计的「产品化」：Google 提供了用户确认敏感操作、检测到间接提示注入时自动停止等企业级保护。这说明 Google 已经意识到，Computer Use 的落地瓶颈不仅是能力，更是信任。PM 在设计 Agent 产品时，应把「可控性」放在和能力同等重要的位置。竞争层面，Gemini 3.5 Flash 的原生 Computer Use 与 OpenAI 的 Operator、Anthropic 的 Computer Use 形成三强对峙，企业 Agent 平台的选择将越来越多地取决于安全、成本、生态整合能力，而不仅是模型智能。

AI Builder 视角：Builder 应将 Gemini 3.5 Flash 的 Computer Use 视为一个「原子能力」，它可以被嵌入到任何需要跨应用操作的 Agent 中。技术上，这意味着你需要重新设计 Agent 的 action space：不再局限于调用 MCP 工具或 API，而是可以直接操作 GUI。这带来了新的挑战：如何定义可执行动作的安全边界？如何处理跨分辨率、跨主题的 UI 变化？如何优雅地失败并回滚？Google 提供了对抗训练和提示注入检测，但 Builder 仍需要在应用层设计沙箱、审计日志、人机协同确认等机制。此外，Flash 系列强调成本和速度，这暗示 Computer Use 的规模化应用需要权衡能力、延迟和成本。

研究型 Builder 视角：从研究角度看，Computer Use 内置到 Flash 模型的最大 novelty 是「通用计算机控制」正在从专门模型能力变成通用模型的子能力。这提示一个关键问题：模型是否正在学习一种跨应用、跨平台的「操作语法」？如果是，那么它的泛化能力将远超传统的 RPA（依赖固定 UI 元素）。研究型 Builder 可关注：1）Gemini 3.5 Flash 在未见过的应用上的 zero-shot 操作能力如何？2）间接提示注入的防御机制是否针对 GUI 场景做了特殊设计？3）与 Gemini 2.5 独立 Computer Use 模型相比，内置版本的性能 trade-off 是什么？这些问题的答案将帮助我们判断 GUI Agent 是否真的进入可大规模部署阶段。

可落地的方法论：- **本周可做**: 如果你正在做企业自动化或 RPA 替代产品，申请 Gemini Enterprise Agent Platform 试用，选取一个 3-5 步的跨应用流程（如从 CRM 导出数据到 Excel 再发送邮件）验证 Computer Use 的端到端成功率。 - **本季度可做**: 设计一套 Agent 操作的安全分级策略，区分只读、可写但需确认、禁止操作三类动作，并将提示注入检测作为默认中间件。 - **长期布局**: 将 Computer Use 与现有工作流引擎结合，构建「自然语言触发 + AI 执行 + 人工复核」的混合自动化产品，重点打磨失败恢复和审计能力。 ---

今天，豆包正式推出专业版

豆包专业版基于豆包 2.1 系列大模型上线，面向复杂办公与生产力场景。办公任务模式接入可执行 Agent 任务的豆包 2.1 模型，支持操作本地电脑、浏览器、调用 Skills 技能、定时任务，内置 Office 办公套件，并可生成带后端数据库的在线应用。定价标准套餐 68 元/月，加强套餐 200 元/月，高级套餐 500 元/月，大学生认证后标准套餐 38 元/月。

AI PM 视角：豆包专业版的发布是字节跳动将 AI 助手从「对话工具」升级为「生产力平台」的关键一步。其核心产品逻辑非常清晰：用 Agent 能力覆盖办公全链路（本地电脑操作、浏览器、Skills、定时任务），用 Office 套件和在线应用生成覆盖内容生产，用分层定价覆盖从个人到企业的不同付费意愿。对 PM 而言，最值得学习的是其「能力打包」策略：它不是卖模型，而是卖一个完整的工作场景解决方案。 定价策略也很有意思：68 元/月的标准套餐低于 ChatGPT Plus 和多数国产 AI 办公产品，大学生 38 元/月的认证价进一步锁定年轻用户。这说明字节正在用价格战快速获取用户基数，同时通过 Skills 生态和高级功能实现 ARPU 提升。竞争层面，豆包专业版直接对标 Kimi、通义、文心一言的付费版，差异化在于「本地电脑操作 + 在线应用生成 + Office 套件」的深度整合。产品成功的关键将取决于 Agent 任务的真实可用性，以及 Skills 生态能否快速丰富。

AI Builder 视角：Builder 可以从豆包专业版中看到一条清晰的工程路线：基础模型（豆包 2.1）+ Agent Runtime（办公任务模式）+ 工具生态（Skills）+ 垂直应用（Office、在线应用生成）+ 定价分层。技术上，「操作本地电脑」需要解决权限、安全、跨应用协调、沙箱隔离等一系列问题；「生成带后端数据库的在线应用」则需要将自然语言需求转化为前端、后端、数据库 schema 的完整工程。Builder 应关注字节如何解决这些工程挑战，尤其是 Skills 的注册、发现、执行和权限管理机制，这可能是未来 Agent 平台的标准参考。

研究型 Builder 视角：从研究角度，豆包专业版的价值在于它将多项 AI 能力（代码生成、RAG、Agent、多模态理解）整合为一个统一产品。研究型 Builder 可以关注：1）豆包 2.1 在 Agent 任务中的规划、执行、反思能力如何评估？2）内置 Office 套件和在线应用生成是否采用了统一的代码生成框架？3）定时任务和本地电脑操作的安全边界如何设计？这些问题的答案将揭示「AI 办公套件」这一产品形态的技术成熟度。

可落地的方法论：- **本周可做**: 如果你是生产力工具 PM，订阅豆包专业版（或申请试用），拆解其办公任务模式、Skills 市场、在线应用生成三个模块的交互流程，输出一份竞品功能对标表。 - **本季度可做**: 在你的产品中设计一个「任务模式」原型，让用户可以用自然语言描述一个多步骤办公任务，由 Agent 自动拆解并执行，重点验证任务完成率和用户满意度。 - **长期布局**: 构建面向特定行业（如电商运营、内容创作、销售管理）的 Skills 生态，通过第三方开发者扩展 Agent 能力边界，形成平台护城河。 ---

火山引擎推出 Agent Ready 基础设施，AgentKit 与 ArkClaw 企业版升级

AI PM 视角：Bidi 1 的上线意味着 ChatGPT 正在从「 turn-based 语音转文字再转语音」的伪语音交互，跃迁到真正的实时双工语音对话。对 PM 而言，这不只是交互延迟的优化，而是产品形态的根本变化：语音交互从「搜索/问答」变成「陪伴、教练、实时协作」的入口。打断能力、边听边说能力、语音中的指令切换能力，直接决定了 AI 能否进入会议、客服、教育、心理咨询、游戏 NPC 等高频场景。

从竞争格局看，OpenAI 此举是对 Anthropic、Google Gemini Live、以及众多端侧语音 Agent 的防御性卡位。语音是移动端最自然的入口，谁掌握了低延迟、可打断、情感自然的双向语音，谁就掌握了下一代 AI 助手的默认调用方式。值得注意的是，Bidi 1 与标准语音、高级语音并列，说明 OpenAI 内部也在多路线并行，尚未确定最终的产品形态。PM 在做语音产品时应避免「一步到位」的幻想，优先验证「可被打断的实时对话」在自身场景中的用户价值。

AI Builder 视角：对 Builder 来说，Bidi 1 的核心工程信号是「端到端双向语音模型」正在成为头部平台的标配。传统的 ASR → LLM → TTS 三阶段流水线，在延迟、情感、打断响应上存在结构性瓶颈。Bidi 1 可能采用了原生音频 token 模型（类似 GPT-4o 的 audio-in/audio-out），将语音直接作为模型的输入输出模态。这意味着 Builder 需要重新评估自己的语音架构：是否还需要独立的 ASR/TTS 供应商？如何处理语音中的背景噪声、多说话人、情绪识别？更关键的是，双向语音对服务端流式工程、客户端音频 buffer 管理、VAD（语音活动检测）提出了极高要求，这些才是产品差异化的真正壁垒。

研究型 Builder 视角：Bidi 1 的技术 novelty 在于它可能将「打断」建模为模型本身的能力，而非通过外部 VAD + 重触发机制实现。如果用户可以在任意时刻插入新指令，模型必须同时具备：持续监听音频流、理解新指令的语义边界、快速切换任务状态、并生成连贯的新回应。这要求模型在音频 token 层面拥有类似文本模型的「上下文切换」能力。一个值得验证的假设是：Bidi 1 是否通过统一的音频-文本联合训练，让模型学会从音频流中识别「打断信号」并自动重置生成？如果是，这将揭示多模态模型在实时交互中的新机制。研究型 Builder 可关注 OpenAI 后续是否释放技术细节，尤其是音频 token 的表示方式、训练数据构造（真实对话 vs 合成打断数据）、以及评估指标（除了 WER，还包括打断响应延迟、任务切换准确率等）。

3 CROSS-CUTTING MODELS
01

框架 1：Agent 能力的产品化漏斗——从「能做」到「敢用」

今天的多个事件共同揭示了一个模式：Agent 技术正在从「能力演示」走向「工程可信」。

  • Bidi 1 让语音 Agent 能被打断，Gemini 3.5 Flash 让 Computer Use 成为标准 API，豆包专业版让 Agent 进入办公套件，火山引擎 AgentKit 提供 Identity/Runtime/Sandbox/Evaluation 四层企业就绪能力。
  • 这个漏斗的上层是「模型能力」（能不能做），中层是「产品封装」（好不好用），底层是「信任基础设施」（敢不敢用）。

产品启示: PM 在设计 Agent 产品时，应将至少 50% 的资源投入到信任基础设施（权限、审计、沙箱、人工复核、失败恢复），而不是一味追求模型能力的领先。企业客户愿意为「可控的 Agent」支付溢价，而不会为「能力更强但不可控的 Agent」承担风险。

02

框架 2：世界模型与 Sim RL——Agent 训练的「数字风洞」

Qwen-AgentWorld 的发布，加上 DFlash 对推理效率的提升，共同指向一个趋势：未来 Agent 的训练和验证将越来越多地在模拟环境中完成。

  • Qwen-AgentWorld 证明世界模型可以在搜索等任务上超越真实环境 RL；DFlash 降低了推理成本，让大规模模拟训练更经济。
  • 这类似于航空业的「风洞测试」：先在模拟环境中验证，再进入真实环境部署。

产品启示: 对于高风险、高成本的 Agent 应用场景（金融交易、工业控制、医疗决策、客服回复），企业应优先投资「数字孪生」或「世界模型」能力，让 Agent 在上线前完成充分演练。这不仅能降低事故风险，还能加速 Agent 的迭代速度。

03

框架 3：垂直 Agent 的「数据-引用-工作流」三角

Perplexity Computer for Counsel、Notion × Cursor、Figma Config 2026 展示了垂直 Agent 的三种落地路径：

  • 数据连接（Perplexity）：连接专业数据库，提供可引用答案。
  • 工作流嵌入（Notion）：将 Agent 嵌入现有协作流程，降低上下文切换成本。
  • 人类判断增强（Figma）：在专业工具中用 AI 增强人类决策，而非替代。

产品启示: 垂直 Agent 的成功不取决于通用模型能力，而取决于对行业数据、用户工作流和专业判断的深度融合。PM 应在进入垂直领域时，优先回答三个问题：我们接入了哪些不可替代的数据源？Agent 如何嵌入用户的日常工作流？人类在哪些环节必须保留最终判断权？

5 DEEP QUESTIONS
01

思考题 1: 当 Agent 可以操作本地电脑和浏览器时，产品的责任边界在哪里？

Gemini 3.5 Flash Computer Use、豆包专业版办公任务模式、火山引擎 AgentKit 同时指向一个未来：Agent 将深度介入用户的本地环境和关键业务系统。这种能力带来了效率飞跃，但也让「误操作」和「恶意利用」的风险急剧上升。

  • 产品策略层面：你的 Agent 产品在哪些操作上必须要求用户确认？哪些可以自动执行？
  • 执行层面：如何设计审计日志和回滚机制，让用户和企业能够追溯和修复 Agent 的行为？
  • 风险与机会层面：责任边界的不确定性，是否会让率先建立透明、可审计 Agent 产品的公司获得信任溢价？
02

思考题 2: 世界模型是否会让「模拟训练」成为 Agent 的标配，还是只适用于特定领域？

Qwen-AgentWorld 证明世界模型在搜索等任务上可以超越真实环境 RL，但它在多大程度上可以泛化到物理世界、金融交易、医疗决策等复杂场景仍不明确。

  • 产品策略层面：你的业务场景中，哪些任务可以用模拟环境替代真实试错？哪些必须依赖真实数据？
  • 执行层面：构建世界模型需要哪些数据资产和工程能力？你的团队是否已具备？
  • 风险与机会层面：如果世界模型成为 Agent 训练的基础设施，数据质量和环境建模能力是否会成为新的护城河？
03

思考题 3: AI 办公套件的竞争，最终是模型之战还是工作流之战？

豆包专业版、Notion × Cursor、Figma Config 2026 代表了三种不同的 AI 办公增强路径。豆包选择自己做全套，Notion 选择集成 Cursor，Figma 选择强化人类判断。

  • 产品策略层面：你的产品应该自建 AI 能力、集成第三方 SDK，还是专注于人机协作体验？
  • 执行层面：如果选择集成，如何设计数据流和用户体验，避免让用户感觉「这是两个产品」？
  • 风险与机会层面：当所有办公工具都具备 AI 能力后，真正的差异化会来自哪里？
04

思考题 4: 在 AI Coding 领域，「可交付性」是否会成为比「代码正确率」更重要的指标？

洪定坤的分享揭示了一个关键现象：主流 Coding 模型组合代码正确率超 80%，但可交付性仅 40-60 分。这说明模型能力与真实工程价值之间存在巨大鸿沟。

  • 产品策略层面：你的 AI Coding 产品是否追踪了可交付性指标？还是只展示了代码生成正确率？
  • 执行层面：需要哪些工程基础设施（测试、CI/CD、代码评审）才能将可交付性提升到 80 分？
  • 风险与机会层面：如果可交付性成为行业共识指标，哪些现有产品会面临重新估值？
05

思考题 5: 当头部 AI 公司开始自研芯片，模型服务的成本结构会如何重构？

OpenAI × Broadcom Jalapeño 加入自研芯片战局，与 Google TPU、Amazon 芯片、NVIDIA 形成复杂竞争。定制推理芯片可能进一步压低推理成本，但也可能加剧生态锁定。

  • 产品策略层面：你的成本模型是否为推理成本持续下降预留了空间？是否过度依赖单一芯片/云供应商？
  • 执行层面：如何设计架构以在多家模型供应商和芯片平台之间灵活切换？
  • 风险与机会层面：推理成本下降是否会催生新的产品形态（如实时语音 Agent、无限上下文）？你的团队是否准备好捕捉这些机会？
