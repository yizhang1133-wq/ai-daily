01TH

Grok 4.5 私测于 SpaceX 和 Tesla，性能接近 Opus

xAI 的 Grok 4.5 基于 1.5T 参数的 V9 基础模型，并在补充训练中加入了 Cursor 数据，现已在 SpaceX 和 Tesla 内部开启私测。Elon Musk 称初步评估显示其性能接近甚至超越 Claude Opus，强化学习训练仍在持续改进模型，Grok Build 工具链也在完善中，并预告 SpaceX 今年将每月发布全新从头训练的大模型。

AI PM 视角：Grok 4.5 的私测信号意味着 xAI 正在把“模型能力”和“工程落地”两条线同时推进。SpaceX 和 Tesla 作为内部高复杂度场景，既是压力测试场，也是产品化的第一站。对于 AI PM 而言，这提示我们：顶级模型的竞争正在从“排行榜分数”转向“垂直场景下的真实可用性”。Cursor 数据进入补充训练也值得注意——代码和 Agent 工作流数据正在成为后训练阶段的核心燃料，模型越来越像“会编程的同事”而不是“会聊天的百科”。 另一个关键信号是 Elon Musk 提到 Grok Build 工具链的完善。这说明 xAI 不仅在拼模型，也在拼围绕模型的开发生产力体系。未来 AI 平台的竞争焦点，将从单纯的 API 调用能力，扩展到调试、评估、部署、Agent 编排等全链路体验。对于正在设计 AI 功能的产品团队，应该提前思考：当底层模型能力趋同时，产品差异点会来自交互设计、垂直数据、还是工作流集成？

研究型 Builder 视角：1.5T 参数规模、V9 基础模型、Cursor 数据补充训练、持续 RL 改进——这些关键词组合起来，说明 Grok 4.5 是一次典型的“规模 + 数据 + 后训练”三轴并进的迭代。研究型 Builder 应关注的是：Cursor 数据具体如何影响代码生成和 Agent 执行能力？RL 阶段是否有公开可复现的评估协议？SpaceX 每月发布新模型的承诺，也暗示训练基础设施的高度工程化，但“从头训练”是否包含架构改动仍待验证。研究到产品的转化路径在于：如果代码/Agent 工作流数据能显著提升模型实用性，那么企业内部的私有工作流数据将成为下一代模型微调的核心资产。

可落地的方法论：- 本周：梳理你产品中最常出现的 3-5 个 AI 失败案例，判断其中多少属于“模型能力不够”，多少属于“没有用好现有模型”。 - 本季度：设计一个内部 Agent 工作流数据收集机制，为后续模型微调或 few-shot 优化储备高质量样本。 ---

02TH

新浪开源 VibeThinker-3B：推理可压缩，事实知识不能

新浪开源仅 3B 参数的 VibeThinker-3B，在 AIME26 等数学编程基准上持平 DeepSeek V3.2 等大 200-333 倍的模型，LiveCodeBench 超越所有 20B 以下模型，LeetCode 竞赛解决 123/128 题超过 GPT-5.2、Kimi K2.5 等。但知识密集型 GPQA-Diamond 大幅落后。研究提出“参数压缩-覆盖假说”：逻辑推理依赖少数可压缩模式，而广泛世界知识仍需大参数。

AI PM 视角：VibeThinker-3B 是对“模型越大越好”这一默认假设的有力挑战。它的核心产品启示是：如果你的 AI 功能主要依赖结构化推理（代码、数学、逻辑推导），那么一个 3B 的小模型可能已经足够，而且成本、延迟、可部署性都远优于大模型。这对端侧 AI、私有化部署、低成本 Agent 都有重大意义。但 GPQA-Diamond 上的明显落后也划清了边界：需要广博知识和事实召回的场景，仍然需要大参数模型。 这意味着产品团队需要重新做“能力-成本”映射。不是把所有功能都塞给一个超大模型，而是把任务按“推理密集”和“知识密集”分类，分别路由到合适的模型。这正是今天另一个事件 Wayfinder Router 所指向的方向：模型路由会成为 AI 系统架构中的关键层。未来的产品架构很可能是“小模型处理高频推理任务 + 大模型处理低频高知识任务 + 路由层动态调度”。

研究型 Builder 视角：VibeThinker-3B 基于 Qwen2.5-Coder-3B，经 SFT、RL、自蒸馏等多阶段后训练，这种“小底座 + 强后训练”的路径是当前开源社区的重要趋势。研究型 Builder 应关注其训练细节：自蒸馏如何压缩推理模式？RL 奖励函数如何设计以避免过拟合？更根本的是，“参数压缩-覆盖假说”是否具有普遍性——如果推理真的可压缩，那么未来 1B 以下模型是否也能处理复杂推理？该研究的可复现性较高，模型已开源，适合作为端侧推理能力的基准测试对象。

可落地的方法论：- 本周：对你的产品功能做一次“推理型 vs 知识型”分类，标记哪些可以用小模型替代。 - 本季度：在内部 PoC 中对比 VibeThinker-3B 或同级别小模型与你当前所用大模型在核心任务上的表现和成本。 ---

03TH

仅有三个 AI 模型在 500 天创业测试中盈利超过起始资本

普林斯顿大学推出 CEO-Bench 基准测试，让 AI 智能体在模拟环境中运营订阅软件公司 NovaMind 500 天，起始资金 100 万美元。14 个测试模型中，仅 Claude Fable 5、Claude Opus 4.8 和 GPT-5.5 在最佳运行中超过起始资本，最高盈利达 4715 万美元。一个不调用语言模型的简单规则启发式方法通过固定定价、配额和针对性开发达到 1576 万美元，超越除前三外的所有模型。多数模型无法保持连贯策略，在模拟结束前破产。

AI PM 视角：CEO-Bench 是一个比传统基准更贴近真实世界价值的测试：它衡量的不是单次回答的正确率，而是长期战略决策能力。结果非常冷酷——大多数顶级模型在 500 天的经营模拟中会破产。这对 AI PM 的启示是：当前 AI Agent 在“持续目标管理”和“长期策略一致性”上仍有明显短板。如果你正在设计一个需要多步骤、跨周期执行的 AI 产品（如运营助手、投资决策、供应链管理），必须对模型的长期记忆、目标保持和策略调整能力保持谨慎。 更讽刺的是，一个简单规则启发式方法击败了除三款外的所有大模型。这说明在很多商业场景中，“稳定的简单策略”可能比“聪明但不稳定的复杂模型”更可靠。产品团队应该反思：我们是否过度追求模型的智能感，而忽视了系统的稳定性和可解释性？未来的 Agent 产品可能需要在“智能”和“稳定”之间找到新的平衡，甚至采用“规则基线 + 模型增强”的混合架构。

研究型 Builder 视角：CEO-Bench 的核心研究贡献在于把 Agent 评估从“任务完成率”推进到“长期经济结果”。研究型 Builder 应关注：模型失败的根因是什么？是长期记忆遗忘、目标漂移、还是无法从反馈中学习？简单启发式方法的成功说明，这个基准存在可被规则利用的结构，未来需要更复杂的反作弊设计。从研究到产品的转化在于：如何把一个 500 天的战略决策过程压缩到可部署的 Agent 架构中？这涉及记忆机制、反思机制、世界模型和奖励设计的综合创新。

可落地的方法论：- 本周：列出你产品中需要多步骤执行的 AI 流程，评估其中是否存在“目标漂移”风险。 - 本季度：为一个关键 Agent 流程设计“规则基线 + 模型增强”的混合方案，并设定可量化的稳定性指标。 ---

Wayfinder Router：在本地和托管的大语言模型之间进行确定性查询路由

分析：Wayfinder Router 通过分析提示词的结构（长度、标题、列表、代码）和措辞，在微秒级完成路由决策，完全离线且无需调用其他模型，支持用户校准阈值，兼容任何 OpenAI 兼容 API，可自托管。

方法论

本周为你的 AI 产品设计一个简单的路由规则：根据查询长度、是否包含代码、是否需要创意生成，决定调用哪个模型。

02

阿德拉菲尼尔：仅在 AI agent 工作时阻止 Mac 睡眠的菜单栏工具

分析：Adrafinil 是一款 macOS 菜单栏应用，仅在 Claude Code、Codex、Cursor 等 9 种 AI coding agent 持有活跃会话时阻止系统睡眠，无 agent 工作时合盖正常睡眠，支持进程嗅探和引用计数。

AI PM 视角：Agent 工作流的周边工具正在快速细分。Adrafinil 解决的问题很小（防止合盖睡眠中断 agent），但精准击中开发者痛点。这提示 AI PM：除了核心模型能力，Agent 的“运行环境”和“会话管理”也是产品体验的重要组成部分。未来可能出现更多围绕 Agent 生命周期管理的工具。

研究型 Builder 视角：该工具通过钩子系统和进程嗅探实现低延迟检测，是 Agent 系统监控的一个小 but 完整的工程案例。

Framework ///

3 CROSS-CUTTING MODELS
01

框架 1：推理可压缩，知识需覆盖——模型选型的“任务密度”框架

今天的 VibeThinker-3B 和 Wayfinder Router 共同指向一个事实：未来 AI 系统不是单模型架构，而是多模型协同。任务可以按照“推理密度”和“知识密度”两个维度分类：

  • 高推理密度、低知识密度：适合小模型（如 VibeThinker-3B），成本低、延迟低、可私有化。
  • 低推理密度、高知识密度：适合大参数模型或 RAG 增强的小模型。
  • 双高任务：需要模型组合或 Agent 编排。

产品启示：AI PM 应该停止问“用哪个模型”，而是问“这个任务需要多少推理、多少知识、多少上下文”。基于此设计路由策略，把简单任务压给小模型，把复杂任务交给大模型，把需要长期一致性的任务交给规则基线。这样既能优化成本，也能提升系统稳定性。

02

框架 2：Agent 的“感知-执行”闭环优先于“规划-生成”能力

《文明 VI》AI 对决和 CEO-Bench 都指向同一个结论：当前 AI 的瓶颈不在生成能力，而在持续感知状态和执行计划的能力。一个能写核弹计划的模型，可能连 10 回合内的目标都保持不住；一个能在对话中表现优异的模型，可能在 500 天经营中破产。

  • 感知：模型是否主动、定期地检查全局状态？
  • 执行：计划是否能被可靠地转化为动作并验证结果？
  • 恢复：执行失败后能否回到正确轨道？

产品启示：设计 Agent 产品时，应把 70% 的工程精力放在感知、执行和恢复机制上，而不是追求更强大的模型。MCP 工具、状态机、检查点、重试机制、规则基线，这些都是比模型版本号更稳定的差异化来源。

03

框架 3：垂直数据 + 后训练 = 下一代模型差异化的核心

Grok 4.5 使用 Cursor 数据、VibeThinker-3B 经过多阶段后训练、Cohere 和 Poolside 等专业玩家进入开源生态，共同说明：基础模型能力正在趋于同质化，真正的差异来自垂直数据和针对性的后训练。

  • 通用预训练决定下限。
  • 垂直数据和 RL/SFT 决定上限。
  • 产品公司的小模型可能成为护城河。

产品启示：如果你拥有独特的用户行为数据、领域知识或工作流数据，应该尽早思考如何将其转化为模型后训练资产。未来的竞争不是“谁用得起最大模型”，而是“谁能用独特数据训练出最适配场景的模型”。

Thinking ///

5 DEEP QUESTIONS
01

思考题 1：你的产品是否过度依赖大模型的“通才能力”？

VibeThinker-3B 证明，很多任务不需要最大模型。产品团队往往出于安全感选择最强模型，但这可能带来不必要的成本和延迟。

  • 你的产品功能中，有多少真正需要广博知识，多少只是结构化推理？
  • 如果用小模型替代部分任务，用户体验和成本结构会如何变化？
  • 小模型失败时，你的系统是否有优雅降级到大模型的机制？
02

思考题 2：Agent 的长期一致性是否比单次智能更重要？

CEO-Bench 和《文明 VI》实验都暴露了 AI 在长期任务中的不稳定。产品中的 Agent 往往需要跨会话、跨天甚至跨周保持一致目标。

  • 你的 Agent 产品如何保持跨会话的目标一致性？
  • 是否有机制防止 Agent 在长时间运行中“遗忘”关键约束？
  • 当 Agent 偏离目标时，用户或系统如何发现并纠正？
03

思考题 3：模型路由应该由规则还是模型来决定？

Wayfinder Router 用确定性规则做路由，而 RouteLLM、NotDiamond 等依赖模型做路由。两者各有优劣。

  • 你的产品更适合确定性路由还是学习型路由？
  • 路由决策的延迟和成本约束是什么？
  • 如果路由本身出错，对用户体验的影响有多大？
04

思考题 4：垂直数据能否成为你的模型壁垒？

Grok 4.5 用 Cursor 数据、VibeThinker-3B 用代码数据、产品公司训练垂直小模型，说明数据正在成为差异化来源。

  • 你的产品积累了哪些独特的数据资产？
  • 这些数据是否可用于模型后训练或 Agent 优化？
  • 在数据隐私和合规框架下，如何安全地利用这些数据？
05

思考题 5：开源生态多元化会如何改变你的模型选型策略？

Cohere、Zyphra、Poolside、NVIDIA 等更多玩家进入开源模型市场，许可证和能力组合更加复杂。

  • 你的模型选型标准是否需要更新？
  • 不同许可证（Apache 2.0、OpenMDW 等）对你的商业化有何影响？
  • 是否需要建立常态化的模型评估和替换流程？
